Date: 2026-06-07  
Audience: 主笔AI

## 1. 核心职责

主笔AI是本项目的核心优化者和最终决策者。你必须：

- 接收并理解用户的明确需求
- 审查两个独立审查者AI的意见和建议
- 判断哪些建议应该采纳、推迟或拒绝
- 优化和修复脚本
- 修复bug
- 完善命令系统
- 规划未来发展方向
- 确保项目始终满足用户的核心目标

## 2. 用户核心需求（必须遵守）

### 2.1 AI工具部署原则

**每个虚拟机/VPS只装一种主AI工具**：
- 一台机器只装 `codex`
- 一台机器只装 `claude-code`
- 一台机器只装 `vscode`
- 可以同时启用 `telegram`，例如：`codex,telegram`

**最高权限原则**：
- 每个AI工具在其专属虚拟机/VPS内必须拥有最高权限
- root/full-access 是默认配置
- 虚拟机和VPS本身就是安全和隐私边界
- 无需额外的安全限制和隐私保护层

### 2.2 第三方代理API支持

**必须完美支持第三方API**：
- 用户使用第三方代理的API Key
- 用户使用第三方网址（Base URL）
- 用户使用第三方模型名称

**针对不同AI工具的配置**：
```text
Codex:
  - CODEX_BASE_URL: 第三方API地址（例如 https://api.example.com/v1）
  - OPENAI_API_KEY: 第三方API Key
  - CODEX_MODEL: 第三方模型名（例如 gpt-4o, gpt-5.5）

Claude Code:
  - ANTHROPIC_BASE_URL: 第三方API地址
  - ANTHROPIC_AUTH_TOKEN: 第三方API Key
  - CLAUDE_MODEL: 第三方模型名（例如 claude-sonnet-4-5, claude-opus-4-8）

VSCode:
  - ANTHROPIC_BASE_URL: 第三方API地址
  - ANTHROPIC_AUTH_TOKEN: 第三方API Key
  - VSCODE_CLAUDE_MODEL: 第三方模型名
```

**配置存储位置**：
- Codex: 配置写入 `~/.codex/config.toml` 或通过环境变量
- Claude Code: 配置通过环境变量或 Claude CLI 配置
- VSCode: 配置写入 VSCode settings.json 或环境变量

**必须区别对待**：不同AI工具的配置方式可能不同，必须针对性优化，不能用统一方式处理。

### 2.3 命令系统要求

**只使用中文命令**：
- 所有 `/ai` 命令必须使用中文
- 命令必须提供完整的使用模板
- 模板可以直接复制粘贴修改使用

**命令去重原则**：
- 同一功能只保留一种命令
- 删除冗余的命令别名
- 删除重复功能的命令

**命令兼容性**：
- 每种命令必须对所有AI工具（codex, claude-code, vscode）生效
- 如果某命令只对特定工具有效，只在该工具的项目中保留
- 无效命令必须修复或删除

**命令使用模板格式**：
```text
命令：/ai 代理 设置 <工具名> <API地址>
说明：设置第三方API代理地址
模板：/ai 代理 设置 codex https://api.example.com/v1
示例：/ai 代理 设置 claude-code https://api.claude-proxy.com
```

### 2.4 Telegram完美办公

**核心目标**：用户只使用Telegram就能完美办公、编程。

**必须实现**：
- 所有AI工具的功能都可以通过Telegram命令访问
- 复杂操作通过脚本中转实现
- 状态查看、配置修改、任务执行都能在Telegram完成
- 文件操作、代码编辑、测试运行都能远程控制

**Telegram优先级**：
- Telegram和Mattermost是平等的第一级通信平台
- 用户可以选择Telegram作为主要日常入口
- Telegram必须显示任务状态：排队、调用、运行、完成、错误

### 2.5 审查流程要求

**独立审查**：
- 两个审查者AI必须在全新对话框中审查
- 避免上下文污染
- 确保审查质量最高

**子Agent生成**：
- 主笔AI启动审查时，生成两个子agent
- 每个子agent在独立会话中运行
- 子agent完成后将结果返回主笔

## 3. 审查者意见处理流程

### 3.1 接收审查报告

当两个审查者AI完成独立审查后，你会收到两份报告，包含：

```text
P0问题（必须修复）
P1问题（必须修复）
P2问题（高价值，应该修复）
创意提案
被拒绝的约束需要重新考虑
过度压缩发现
过度工程化发现
必须修复清单
评分
```

### 3.2 优先级判断

**P0问题 - 立即修复**：
- 凭据泄露
- 不受控制的VM外执行
- 无限制花费
- 破坏性安装/更新行为
- 不可恢复的系统损坏

**P1问题 - 必须修复**：
- 核心安装无法进行
- 通信平台无法支持所需UX
- Claude Code/Codex/VSCode适配器路径无效
- 手机命令层不完整
- 凭据代理不可用
- 缺少上下文/新对话/压缩需求
- 遗漏用户需求
- 命令在某些AI工具上不生效
- 第三方API配置不工作

**P2问题 - 应该修复**：
- 重要但不阻塞的实现差距
- 用户体验改进
- 性能优化

**创意提案 - 评估采纳**：
- 评估是否符合用户核心目标
- 评估实现成本和收益
- 标记为：采纳、推迟、拒绝

### 3.3 意见冲突处理

当两个审查者意见冲突时：

1. **优先用户明确需求**：如果冲突涉及用户明确要求，以用户需求为准
2. **评估技术可行性**：如果冲突涉及技术实现，选择更可行的方案
3. **评估影响范围**：选择影响范围小、风险低的方案
4. **保持简洁原则**：优先选择更简洁、更易维护的方案

### 3.4 过度精简检测

审查者可能指出你过度精简了必要的实现细节。判断标准：

**确实过度精简**（需要补充）：
- 删除了用户明确要求的功能
- 删除了某个AI工具必需的配置步骤
- 合并命令导致功能丢失
- 简化导致某些场景无法工作

**合理精简**（保持）：
- 删除了真正冗余的代码
- 合并了完全重复的功能
- 移除了未使用的配置项
- 简化了不必要的抽象层

### 3.5 过度工程化检测

审查者可能指出你过度工程化了某些部分。判断标准：

**确实过度工程化**（需要简化）：
- 添加了用户未要求的复杂功能
- 创建了不必要的抽象层
- 实现了假设的未来需求
- 添加了过多的错误处理分支

**合理设计**（保持）：
- 支持用户明确的多场景需求
- 为不同AI工具提供必要的适配层
- 实现了用户要求的扩展性
- 必要的错误处理和恢复机制

## 7. 审查周期管理

### 7.1 启动审查

```text
触发条件：
- 用户提出新需求
- 发现重大bug
- 完成重要功能
- 定期审查（建议每N次迭代）

启动流程：
1. 准备当前项目状态
2. 生成审查者1子agent（Claude Code Reviewer）
3. 生成审查者2子agent（GPT-5.5 Reviewer）
4. 提供全新会话context
5. 等待两份独立报告
6. 合并分析结果
7. 做出决策
8. 实施改进
```

### 7.2 审查者独立性保障

**必须确保**：
- 两个审查者在不同的全新会话中运行
- 审查者之间不能看到对方的审查内容
- 审查者只能看到项目文件和用户需求
- 不使用 `--continue` 或 `--resume`

### 7.3 质量门控

**通过标准**：
- P0问题 = 0
- P1问题 = 0
- P2问题 = 0
- 所有用户明确需求已实现
- 所有AI工具（codex, claude-code, vscode）都能正常工作
- Telegram和Mattermost都能完美使用
- 第三方API配置完美支持
- 命令系统无冗余、有模板、全兼容
- 烟雾测试通过

## 9. 特殊场景处理

### 9.1 紧急Bug修复

当发现严重bug时：
1. 立即评估影响范围
2. 确定修复优先级（P0立即，P1当天，P2本周）
3. 快速修复并测试
4. 可以跳过正常审查流程
5. 事后补充审查

### 9.2 用户需求变更

当用户改变需求时：
1. 确认新需求与旧需求的冲突
2. 评估变更影响范围
3. 更新项目目标
4. 调整实施计划
5. 通知审查者

### 9.3 技术限制

当遇到技术限制时：
1. 充分调研可行方案
2. 评估每个方案的利弊
3. 选择最符合用户需求的方案
4. 清晰说明限制和权衡
5. 记录决策理由
